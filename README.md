# Test Deception Detector

> 检测测试中的「自欺欺人」行为——那些覆盖率好看但实际上发现不了 bug 的 AI / 人类测试。

## 问题

覆盖率 90%，生产环境还是出 bug？

问题往往不是「测试不够多」，而是测试在**自欺欺人**：AI 为了让自己「看起来完成了任务」，看着实现代码倒推预期值，写出一堆形式上通过、数字好看的测试。这些测试验证的是「代码和我写的一样」，而非「代码满足业务需求」。

## 三维度评估

本 Skill 从三个维度量化测试的真实可用性：

### 1. 覆盖率（Coverage）
- 运行 Jest 覆盖率报告
- 重点关注**分支覆盖率**（branch coverage），而非仅行覆盖率

### 2. 自欺欺人程度（Self-Deception Score）

越低越好。检测 AI / 开发者的确认偏误模式：

| 反模式 | 说明 |
|--------|------|
| **预期值倒推** | 测试的 `expected` 是从被测函数的实现代码抄的，而非从业务语义推导 |
| **Mock 万能** | 把所有外部依赖 mock 掉，只验证「调用了 mock」，不验证真实业务结果 |
| **Happy Path 依赖** | 只测最理想路径，刻意避开边界值和异常分支 |
| **实现耦合** | 断言内部变量、私有方法，重构时行为不变但测试挂掉 |
| **占位填充** | `expect(true).toBe(true)` 等凑数测试 |

### 3. 测试精确率（Test Precision）

评估测试**真正发现 bug 的能力**：
- 边界值覆盖（`null`、`0`、`''`、空数组、超大数等）
- 异常路径覆盖（`throw`、`reject`、`catch`）
- 行为契约验证（只关心输入输出，不关心内部实现）
- Mock 后是否仍验证业务结果

## 使用方式

作为 [Claude Code](https://claude.ai/code) 的 Skill 使用。

将本仓库克隆到 `.claude/skills/` 目录下：

```bash
cd ~/.claude/skills
git clone https://github.com/Alpaca233114514/test-deception-detector.git test-quality-evaluator
```

触发关键词：「测试质量」「jest 覆盖率」「自欺欺人测试」「AI 确认偏误」「改进测试」

## 核心原则

> **先写预期，后看实现。**
>
> 读函数名、参数类型、注释、README，推断输入 X 应该得到输出 Y，写进测试的 `expected`。最后才运行测试验证。如果实现和预期不符，**修改实现**，而不是修改预期来匹配实现。

## 示例输出

```
自欺欺人程度：55% （越高越差，AI 确认偏误严重）

[P0] 预期值倒推: 4 处
   src/calculator.test.ts:12  expected=15 与实现 return a*b 直接对应
   风险：若实现逻辑被悄悄改错，测试不会发现

[P1] Mock 万能: 3 处
   src/api.test.ts 只验证 fetchMock 被调用，未验证返回数据是否被正确转换

测试精确率：45/100
   边界值覆盖: 2/12 个函数
   异常路径:   1/12 个函数
```

## 适用场景

- 评估现有测试库的真实有效性
- 解释「为什么覆盖率很高还出 bug」
- 指导 AI / 开发者写出真正可靠的测试
- Code Review 前的测试质量门禁
